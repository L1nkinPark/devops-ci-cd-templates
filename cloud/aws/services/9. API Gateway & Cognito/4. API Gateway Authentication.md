# Authentication cho API Gateway

API Gateway cung cấp 2 phương thức xác thực tích hợp trực tiếp (authorizer) thường được sử dụng nhất để bảo mật các endpoint:

## 1. Cognito Authorizer

Phương thức này liên kết trực tiếp API Gateway với một **Cognito User Pool** để sử dụng làm authorizer.

- **Cách hoạt động:** Khi muốn truy cập API, client sẽ đính kèm (pass) trực tiếp token nhận được sau khi đăng nhập thành công qua Cognito vào request. API Gateway sẽ tự động kiểm tra token đó và cho phép (allow access) nếu token hợp lệ.
- **Ưu điểm:** Dễ dàng cấu hình và tích hợp sẵn mà không cần lập trình thêm logic xác thực.

## 2. Lambda Authorizer (Custom Authorizer)

Khi sử dụng loại authorizer này, bạn sẽ tự lập trình (implement) logic xác thực tùy chỉnh của riêng mình trên một hàm AWS Lambda.

Có 2 hình thức Lambda Authorizer chính:
- **Token-based:** Xác thực dựa vào một Token (thường là JWT hoặc OAuth token) được gửi trong Header.
- **Request parameter-based:** Xác thực dựa vào các tham số khác có trong request (ví dụ: Query parameters, Headers chứa username/password...).

### Sơ đồ hoạt động của Lambda Authorizer

![Lambda Authorizer Architecture](../../../../images/aws/api_gateway_lambda_auth.png)

**Quy trình (Workflow):**
1. **Client** gửi request kèm theo Bearer token hoặc các tham số đến **API Gateway**.
2. API Gateway gọi hàm **Lambda Auth** và truyền các thông tin xác thực này sang.
3. Hàm Lambda xử lý logic kiểm tra và trả về một IAM Policy (quy định Allow/Deny) cùng định danh người dùng (Principal).
4. API Gateway đánh giá Policy nhận được:
   - Nếu **Allowed**: Cho phép chuyển tiếp request đến hệ thống Backend (AWS Lambda, EC2...).
   - Nếu **Denied**: Trả thẳng lỗi `403 Forbidden` về cho Client.
5. *(Tùy chọn)* API Gateway có thể **cache** lại Policy này để giảm độ trễ (latency) cho các request tiếp theo từ cùng một Client.
